## 0. 핵심 결론

- **데이터 원장은 이미 하나다.** `sales_orders` + `sales_order_items`가 단일 진실원장이며 카페24·배송·매출·재고가 모두 여기에 연결된다. DB 재구축 불필요.
- 문제는 **화면·메뉴 분산 + 상태축 3종 혼재**. 해결 = "같은 전표의 목적별 뷰로 재정렬 + 상태모델 2축 통일".
- **배송 메뉴 전면 흡수**(사용자 확정): 최상위 `배송` 제거 → `판매관리` 단일 진입, 5탭.

## 2. 상태값 통합 모델 (2 master 축 + 디테일 강등)

현재 `receipt_status` ↔ `shipment.status` ↔ `cafe24 order_status` 3축 혼재가 버그 근원(#18·#19·#25·#32~38). → **전표는 2축만, 나머지는 디테일.**

### 축① 수령상태 `receipt_status` (전표 master)
```
방문:  PICKUP_PLANNED(방문예정) ───────────────► RECEIVED(수령완료)
퀵:    QUICK_PLANNED(퀵예정) ─────────────────► RECEIVED(배송완료)
택배:  PARCEL_PLANNED(택배예정) → PARCEL_SHIPPED(택배발송) → RECEIVED(배송완료)
```
- 표시 라벨: 택배·퀵의 RECEIVED = **배송완료**, 방문 = **수령완료** (채널별 라벨, 내부값 RECEIVED 동일 — 직전 적용분).

### 축② 결제상태 `approval_status` (전표 master)
```
COMPLETED(결제완료) / CARD_PENDING(미승인) / UNSETTLED(미결)
```

### 디테일(전표 master 아님 — 특정 화면에서만)
| 디테일 | enum | 노출 화면 | 전표 연동 |
|---|---|---|---|
| 배송 상태 | shipments PENDING→PRINTED→SHIPPED→DELIVERED | **택배관리만** | SHIPPED→PARCEL_SHIPPED, DELIVERED→RECEIVED 자동(#19) |
| 카페24 상태 | 입금전/결제완료/배송중… | **온라인몰만** | 확정(배송추가) 시 전표로 승격(#25) |

★ **택배관리 표시 단순화(사용자 확정)**: 4단계 배송상태를 **2버킷**으로 묶어 본다.
  - **대기중** = PENDING (미발송)
  - **발송완료** = PRINTED · SHIPPED · DELIVERED (출력·발송·배송 = 동일 행위로 간주)
  - **전체** = 모두(추적 포함). 내부 enum 4값은 유지하되 화면 필터·집계만 2버킷.

**판매현황은 ①②만 본다.** 송장 디테일은 택배관리, 몰 원천상태는 온라인몰에서만 → 같은 주문을 다른 상태로 중복 확인할 일 제거.

## 3. 메뉴 구조 (사용자 확정 IA 2026-06-18)

### 최상위 7개 그룹 (현 15개 → 축소)
```
[판매]   [고객]   [재고]   [생산]   [구매]   [제품]   [회계]
```
- 매핑: 판매관리→**판매**, 고객관리→고객, 재고→재고, 생산→생산, 매입→**구매**(개명), 제품→제품, 회계→회계.
- 배송(흡수됨)·대시보드·거래관리·알림·코드·보고서·AI메모리·AI대화기록 = §3-1 결정 대기(삭제 아님, 처리방식 미정).

### [판매] 하위 구조
```
판매
 ├─ 판매입력      오프라인/지점/직접 판매 입력        (현 checkout)
 ├─ 판매현황      전체 확정 전표 조회·수정·수령상태    (현 SalesListTab)
 ├─ 온라인몰      카페24 수집·고객매칭·품목매핑·확정  (현 /shipping > 카페24주문)
 ├─ 택배관리      택배 전표 발송 처리                  (현 /shipping > 배송목록)
 │   ├─ 전체       (추적 포함 가능)
 │   ├─ 대기중     미발송 = shipment PENDING / receipt PARCEL_PLANNED
 │   └─ 발송완료   ★ 출력·발송·배송을 **동일 행위(발송완료)**로 간주
 │                  → shipment PRINTED·SHIPPED·DELIVERED 를 한 버킷으로 묶음
 └─ 매출현황      전표 기준 집계 (현 /reports > 매출)
     └─ 일 / 월 / 년
```
- 명칭 확정: 판매등록→**판매입력**, 온라인몰관리→**온라인몰**, 매출관리→**매출현황**.
- `직접 배송 입력`은 **축소**(예외 입력만, 기본 동선 제외).
- 라우트: 1차 진입 통합(임베드), 점진 이동. `screen_permissions` 갱신.

### 3-1. 미결 — 핵심 7개 외 기존 메뉴 처리 (결정 대기)
대시보드 · 거래관리(B2B) · 알림 · 코드 · 보고서(매입/손익/트렌드/마진) · AI메모리 · AI대화기록.
→ (A) 하단 '관리/기타' 그룹으로 분리 보존 / (B) 숨김(권한·직접URL만) / (C) 상단 7개만 우선정렬+나머지 잔존.
