# AI 에이전트 전략 검토 — 직원 대체 가능성 심층 분석

## 1. 출발점: 원래 구상

시스템이 DB 스키마와 프로그램 구조를 미리 알고 있으므로, 일반 직원보다 능숙하게 업무를 처리할 수 있다는 가정.
장기적으로는 직원을 채용한 것과 동등한 수준으로 에이전트가 업무를 학습·진화하여 사람 없이도 회사를 운영하는 것이 이상적 목표.

---

## 2. 현재 시스템 위치 (객관적 스냅샷)

- `src/lib/ai/tools.ts` — 약 **41개 도구** / 2,164줄
- `src/lib/ai/schema.ts` — DB 스키마 요약 67줄
- `src/app/api/agent/route.ts` — 376줄 (툴 콜링 라우트)
- `src/app/(dashboard)/agent-memory` — 장기 메모리 페이지
- 구조: Claude가 스키마와 화이트리스트 도구를 알고 있는 **툴 콜링(tool-use) 에이전트**

**이미 달성한 것**: L1(조회/집계), L2(단순 CRUD) 업무는 사람보다 빠르고 정확함 — 이건 과장이 아님.

---

## 3. "직원 대체"의 5단계 분해

직원 1명이 실제로 하는 일을 다섯 층으로 나누면 각 층마다 LLM 에이전트의 현실적 성취도가 다름.

| 층 | 예시 업무 | 현재 기술로 달성도 |
|---|---|---|
| L1 **조회/요약** | "이번 달 경옥고 재고 얼마?" | ✅ **95%** — 이미 잘 됨 |
| L2 **단순 실행** | "이 주문 환불 처리해줘" | ✅ **85%** — 도구만 있으면 됨 |
| L3 **다단계 판단** | "이 환불 요청 타당한지 보고 처리" | ⚠️ **55%** — 규칙 명시되면 가능, 경계 케이스 위험 |
| L4 **이해관계 조율** | "공급사와 납기 협상", "불만 고객 달래기" | ❌ **20%** — 텍스트는 그럴싸해도 책임·관계는 사람이 져야 함 |
| L5 **창의·전략** | "내년 매출 20% 올릴 방안 실행" | ❌ **10%** — 제안은 가능, 실행·검증은 사람 필수 |

**결론**: L1~L2는 지금 당장 사람을 **대체** 가능. L3는 **보조** 가능하지만 완전 대체는 리스크. L4~L5는 현재 기술로 무리.

---

## 4. "스스로 학습·진화하는 에이전트" — 냉정한 검증

가장 오해가 많은 영역. 정직한 판단:

**LLM은 본질적으로 스스로 학습하지 않는다.** Claude/GPT 모델의 가중치는 추론 중 변하지 않음. "학습한다"고 부를 수 있는 건 세 가지뿐:

### 4-1. 메모리 축적 (agent-memory 같은 방식)
- 과거 대화/결정을 텍스트로 저장 → 다음 호출에 context로 주입
- **한계**: context window만큼만 기억. 그 이상은 검색(RAG)으로 요약
- 이건 "학습"이 아니라 **"서류 캐비닛이 커지는 것"**

### 4-2. Fine-tuning (주기적 재학습)
- 회사 데이터로 모델을 재학습
- **한계**: 비용·주기·품질 통제 어려움. 작은 회사가 운영하기엔 과함
- 현실적으로 대부분의 업무용 에이전트는 하지 않음

### 4-3. Self-improvement loop (AutoGPT류 자율 개선)
- 에이전트가 자기 프롬프트·도구를 스스로 바꾸는 시도
- **현실**: 2024~2025년 현재 성공 사례 거의 없음. 주기적으로 엉뚱한 방향으로 탈주. 감독 비용이 절약된 인건비보다 큼

### 냉정한 판단
**"스스로 학습하고 진화하는 에이전트"는 기술 마케팅 표현이지 현실 제품이 아님.** 앞으로 3~5년 안에 회사 단위에서 구현 가능하지 않음.

대신 **"진화처럼 보이는 것"**은 만들 수 있음 — 사람이 피드백을 주고, 그게 메모리/규칙/도구 개선으로 반영되는 **반자동 학습 루프**. 이게 현실적으로 유일하게 작동하는 방식.

## 5. 직원 대체가 실패하는 5가지 현실적 이유

1. **책임 귀속** — 에이전트가 잘못된 환불을 처리하면 누가 책임지나? 법·세무·회계는 사람 결재가 필수. 금액 임계치 이상은 자동화 금지가 현실적
2. **경계 케이스 판단** — "이 고객은 컴플레인 이력이 있으니 정책 외 환불 승인" 같은 맥락. 도구로 학습 안 됨
3. **물리적 액션** — POS에서 돈을 만지고, 택배를 포장하고, 상담 전화를 받는 일은 여전히 사람
4. **신뢰 구축** — 단골 고객은 에이전트가 아니라 "박 과장님"과 얘기하고 싶어함. 이건 UX가 아니라 사회적 현상
5. **Hallucination** — LLM은 여전히 없는 사실을 만들어냄. 100건에 1건 틀려도, 그 1건이 1억짜리면 전체 시스템 불신으로 이어짐

---

## 6. 현실적 목표 재설정

**"직원 대체"가 아니라 "직원 1명당 처리량 3~5배 + 대표 인지부하 절감"**

### Phase 1 — 현재~6개월 (보조 에이전트)
- 도구 41개 → **70~80개** 확장. 특히 L2 "단순 실행" 두텁게
- 모든 에이전트 실행에 **사용자 확인 단계**(confirm) 강제 — L3 이상은 반드시 사람 재가
- `agent_action_logs`로 **감사 로그** 필수 축적
- 직원은 에이전트에게 "시키는" 역할로 전환

### Phase 2 — 6~18개월 (반자동 에이전트)
- 감사 로그를 주기적으로 사람이 라벨링 → "옳은 판단" / "틀림"
- 라벨링 결과를 프롬프트/규칙/메모리에 반영 ← **"진화처럼 보이는 것"의 실체**
- 루틴 업무(매일 반복)는 **cron + 에이전트**로 무인화
  - 예: 매일 09시 카페24 주문 동기화 → 이상값 있으면 알림톡 → 사람이 승인
- 직원은 "예외 처리 + 에이전트 감독자"로 전환 — 이 시점에 실제 인력 절감 발생

### Phase 3 — 18개월+ (일부 직무 자율화)
- 저위험·반복·경계 명확한 직무는 **특정 시간대 무인 운영**
- 예: 자사몰 주문 접수 + 고객 문의 1차 응답 + 재고 알림 + 송장 자동 발번
- 그러나 **고위험·고금액·고감성 업무는 여전히 사람** — 영구적으로

## 7. 최종 판단 (개인 의견)

"에이전트가 직원을 완전히 대체"라는 목표는 **잘못된 프레임**.

### 이유 세 가지

**첫째, 목표가 틀리면 투자가 왜곡된다.**
"직원 대체"를 목표로 두면 에이전트에게 자율권을 과하게 주고, 감사·확인 단계를 줄이게 되고, 결국 한 건의 큰 사고로 전체 시스템 신뢰가 무너짐. 반대로 "직원 증강"을 목표로 두면 안전하게 쌓을 수 있고, 실패해도 복구 가능.

**둘째, 현재 기술 수준에서 "자율 에이전트"는 감독 비용이 절약 비용보다 크다.**
여러 자율 에이전트 프로젝트에서, 사람이 결과를 검증하는 시간이 원래 직접 하던 시간보다 많아지는 경우가 태반. 중소 규모 회사에서 이건 투자 대비 손해.

**셋째, 진짜 가치는 "전환"이 아니라 "증폭"이다.**
이 시스템의 잠재력은 직원을 없애는 게 아니라, **직원 채용 없이 매출을 2~3배 키울 수 있게 하는 것**. 5명 회사가 에이전트로 15명 분 업무를 처리하면, 그게 훨씬 실용적 승리.

### 희소식

현재 시스템 구조는 이미 올바른 방향. DB 스키마 인식, 도구 화이트리스트, 메모리 테이블, 감사 로그 같은 기초는 다 갖춤. 부족한 건 기술이 아니라 **운영 규율**:

1. 모든 에이전트 작업에 승인 단계를 둘 것
2. 모든 작업을 로그화하고 매주 10건씩이라도 사람이 검수할 것
3. 검수 결과를 메모리/규칙/도구에 되먹일 것

이 루프가 3~6개월 돌면, "진화한다"는 느낌을 실제로 받을 수 있음. LLM이 진화한 게 아니라, **시스템과 대표가 같이 진화한 것**. 그게 가장 빠르고 안전한 길.

---

## 8. 결론 요약

- **기술적으로 "스스로 학습하고 진화하는 에이전트"는 환상에 가까움** (3~5년 내 불가)
- **그러나 "직원 증강"은 지금 당장 실현 가능이고 ROI 확실함**
- 현재 시스템 기반은 이미 좋은 방향. 필요한 건 **운영 규율과 피드백 루프**
- 목표를 "직원 대체"에서 **"직원 1인당 처리량 3~5배 + 대표 인지부하 절감"**으로 재설정 권고
