# 경옥채 사내 통합시스템 — QA 테스트 플랜

**버전**: 2026-04-08
**대상**: 전 시스템 화면 20개 + AI 에이전트 + 통합 시나리오
**목적**: 실사용 배포 전 기능·권한·통합 검증

---

## 0. 사용법

### 0-1. 테스트 절차
1. 테스터가 지정된 계정으로 로그인
2. 화면별 테스트 케이스를 순서대로 실행
3. 각 케이스의 **기대 결과**와 실제 결과를 비교하여 결과란에 ✅/❌/⚠️ 기록
4. ❌ 발생 시 재현 경로·스크린샷·콘솔 에러를 별도 기록
5. 화면별 섹션 끝의 총평란에 소요 시간·총 성공률 기록

### 0-2. 표기 규칙
- ✅ 정상 / ❌ 실패(재현 가능) / ⚠️ 부분 동작(버그 있으나 우회 가능) / ⏭️ 스킵(권한/데이터 없음)
- 심각도: **P0**(즉시 수정) / **P1**(다음 스프린트) / **P2**(개선)

### 0-3. 테스트 계정 준비
각 역할별 테스트 계정을 먼저 생성 (아직 없다면 `/login`에서 관리자가 추가):

| 역할 | 계정명 예시 | 담당 지점 |
|---|---|---|
| SUPER_ADMIN | qa_super | (없음) |
| HQ_OPERATOR | qa_hq | (없음) |
| EXECUTIVE | qa_exec | (없음) |
| PHARMACY_STAFF | qa_pharm | 한약국 지점 1개 |
| BRANCH_STAFF | qa_branch | 백화점/자사몰 지점 1개 |

## 5. 통합 시나리오 (End-to-End)

### 5-1. POS 판매 → 환불 전체 흐름
1. `/pos`에서 상품 2개 장바구니 → 고객 회원 선택 → 현금 결제
2. 영수증 모달 확인 → 전표번호 기록
3. `/customers/[id]`에서 고객 포인트 적립 확인
4. `/reports` → 해당 날짜 매출 반영 확인
5. `/accounting` → 매출 분개 생성 확인
6. `/pos` → 환불 모달 → 고객·날짜 검색 → 해당 주문 클릭 → 부분 환불 처리
7. `/reports`·`/accounting`에 역분개/수치 반영 확인

### 5-2. 발주 → 부분 입고 → 재입고
1. `/purchases`에서 발주 생성 (A 제품 100개)
2. 확정 (CONFIRMED)
3. AI에게 "A 제품 30개만 입고해줘" → PARTIALLY_RECEIVED + 재고 +30
4. 수동으로 나머지 70개 입고 → RECEIVED
5. `/inventory` 재고 +100 확인

### 5-3. 생산지시 → 원재료 차감 → 완제품 입고
1. BOM이 등록된 제품으로 생산지시서 생성
2. Start → 원재료 충분 확인
3. 완료 → 원재료 감소 + 완제품 증가 확인 (inventory_movements 기록)

### 5-4. 카페24 주문 전체 흐름 (Webhook 없는 환경)
1. `/shipping` 카페24 탭 → 토큰 갱신
2. 결제완료 매출 동기화 (기간 지정)
3. `/reports` → 자사몰 채널 수치 증가 확인
4. 카페24 주문 탭에서 선택 → 배송 등록
5. 대한통운 엑셀 다운로드 → (실제 CJ 전산 업로드 생략) → 송장번호 엑셀 임포트
6. 배송상태 일괄 업데이트 → SHIPPED 전환 확인
7. AI에 "오늘 배송 건 조회" → get_shipments 결과 확인

### 5-5. 카페24 회원 동기화 → 고객 식별
1. `/customers`에서 "카페24 회원 동기화" 클릭
2. 신규 건수 확인
3. `/customers`에서 임의 회원 열람 → `cafe24_member_id` 표시 확인
4. 카페24 신규 주문 webhook 수신 시 해당 회원과 매칭되는지 로그 확인
